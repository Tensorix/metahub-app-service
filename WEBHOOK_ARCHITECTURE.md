## 数据流

### 1. 请求阶段 (同步)

```
外部系统 → Router → 认证验证 → 返回 202 Accepted
                      ↓
                 添加后台任务
```

**时间**: < 100ms

### 2. 处理阶段 (异步)

```
后台任务启动
    ↓
创建/更新 Session (基于 external_id)
    ↓
创建/更新 MessageSender
    ↓
创建 Message + MessagePart
    ↓
获取上下文 (最近 30 条消息)
    ↓
创建 Event
    ↓
调用 LangChain Agent
    ↓
LLM 分析 (OpenAI API)
    ↓
解析结构化输出
    ↓
如果重要 → 创建 Activity
    ↓
提交数据库事务
    ↓
记录日志
```

**时间**: 2-5 秒 (取决于 LLM 响应速度)

## 性能优化策略

### 1. 异步处理
- 使用 FastAPI BackgroundTasks
- 立即返回 202 Accepted
- 后台处理不阻塞请求

### 2. 数据库优化
- external_id 字段添加索引
- 使用 flush() 而非 commit() 在事务中
- 批量查询减少数据库往返

### 3. LLM 调用优化
- 使用 temperature=0.1 提高一致性
- 限制上下文消息数量 (30 条)
- 可选: 添加 Redis 缓存相似消息

### 4. 错误处理
- try-catch 包裹所有操作
- 失败时返回默认值 (不重要)
- 详细日志记录便于调试

## 安全考虑

1. **认证**: 支持 JWT 和 API Key 双认证
2. **用户隔离**: 所有数据都关联 user_id
3. **输入验证**: Pydantic 模型验证所有输入
4. **错误处理**: 不暴露内部错误信息
5. **日志脱敏**: 避免记录敏感信息
6. **速率限制**: 建议添加 rate limiting
7. **HTTPS**: 生产环境必须使用 HTTPS

## 总结

这个架构设计具有以下优点：

✅ **高性能**: 异步处理，不阻塞请求
✅ **可扩展**: 模块化设计，易于扩展
✅ **智能化**: LangChain Agent 自动判断
✅ **可靠性**: 完善的错误处理
✅ **安全性**: 双认证 + 用户隔离
✅ **可维护**: 清晰的分层架构
✅ **可观测**: 详细的日志记录
